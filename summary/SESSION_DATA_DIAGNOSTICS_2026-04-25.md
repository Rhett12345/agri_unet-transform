## Current Aggregation Implementation

Core aggregation is in:

```text
fusion_core.py::aggregate_modis_to_agri()
fusion_core.py::_collect_1km()
```

Current behavior:

1. AGRI lat/lon grid is flattened.
2. Valid AGRI pixels are converted to unit-sphere XYZ.
3. Each MYD06 granule provides `CLP_1km`, `CER_1km`, `COT_1km`, `CTH_1km`.
4. MYD03 `lat_1km/lon_1km` is preferred.
5. If missing, code can upsample 5 km geolocation to 1 km, but current strict fusion diagnostics show this fallback is not being used.
6. MYD03 EV start time provides `scan_time_1km`.
7. MODIS 1 km points are put into a `cKDTree`.
8. For each AGRI pixel, `query_ball_point()` collects all MODIS 1 km pixels within `AGRI_SEARCH_RADIUS_KM`.
9. Current default radius is `3.0 km`.

Per-AGRI-pixel aggregation:

- `CLP`: weighted majority vote.
- `PHASE_CONSISTENCY`: dominant class weight divided by total class weight.
- `COT`: log-domain weighted median using cloudy candidates only.
- `CER`: weighted mean using cloudy candidates only.
- `CTH`: weighted mean from `CTH_1km`.
- `OVERLAP_FRACTION`: `n_1km / EXPECTED_1KM_PER_AGRI`, clipped to 1.0.
- `EXPECTED_1KM_PER_AGRI` is currently 16.

Important implementation note:

- Some comments still mention separate 1 km vs 5 km aggregation, but current code aggregates `CTH_1km` inside `_collect_1km()` with the other 1 km fields.
- `VALID_PIX_5KM` naming is now misleading in the current implementation because it is assigned from `len(cth_v)`, not from an independent 5 km collection.

## Main Technical Conclusion

The current evidence points away from:

- file-level scan-time fallback,
- MYD06 5 km geolocation fallback,
- patch row/col slicing errors,
- one fixed global spatial shift,
- `apply_quality_filter()` creating the problematic patterns.

The current evidence points toward:

- label noise or physical inconsistency introduced during MODIS-to-AGRI aggregation,
- water-class supervision being especially weak,
- radius-based KD-tree aggregation creating mixed/fragmented labels near cloud and phase boundaries,
- regression labels being too sparse and strict relative to the classification labels,
- possible need to change data filtering or fusion thresholds before any model changes.
